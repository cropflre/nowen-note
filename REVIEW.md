# NowenNote 项目全面审查与优化方案

> 审查日期：2026-06-04
> 项目定位：面向家庭 3-4 人用户的笔记/说说/待办协作平台（二次开发自开源 NowenNote）
> 审查范围：产品功能、技术架构、UI/UX 三大维度

---

## 一、产品功能全景梳理

### 1.1 现有模块清单

| 模块 | 核心能力 | 面向场景 |
|------|---------|---------|
| **笔记** | 富文本/Markdown 双编辑器、笔记本树、标签、附件、版本历史、全文搜索、回收站 | 个人知识管理、写作 |
| **说说（日记）** | 时间线式短内容、心情表情、图片&语音、公开/私密可见性、语音转文字 | 生活记录、社交轻动态 |
| **待办事项** | 任务层级（父子）、优先级、截止日期、提醒、标签、日历视图 | 个人/GTD 任务管理 |
| **思维导图** | 节点画布、缩放/编辑 | 头脑风暴、大纲整理 |

| **文件管理** | 附件聚合视图、网格/列表、分类筛选、孤儿清理 | 全局文件管理 |
| **工作区** | 团队空间、成员角色（owner/admin/editor/commenter/viewer）、功能开关 | 家庭/小团队共享 |
| **分享** | 笔记分享链接、密码保护、过期时间、评论权限 | 对外内容分享 |
| **导入** | 小米云/OPPO 云/iCloud/有道/Memos/URL/Markdown/Word 多源导入 | 迁移其他平台数据 |
| **安全** | JWT 认证、2FA TOTP、API 令牌、会话管理、登录锁定 | 账号安全 |
| **实时协作** | Yjs CRDT、WebSocket 协同编辑、在线感知 | 多人同时编辑 |

### 1.2 数据实体关系

```
用户 ──┬── 笔记本 ── 笔记（FTS5 搜索 + 版本历史 + 附件）

       ├── @提及通知
       └── 分享链接 ── 评论
```

## 二、产品层优化建议

### 2.1 家庭场景核心缺失

#### 🔴 问题 1：没有「家庭仪表盘」首页

当前没有默认首页。登录后进入 `viewMode === "all"` 的笔记列表。对于家庭用户，应该有一个聚合仪表盘：

- 家庭成员的最近说说（类似微信朋友圈入口）
- 共享待办（"买牛奶"、"交水电费"）
- 家庭笔记（食谱、WiFi 密码、装修记录）
- 最近活动的时间线

**建议：** 新增「家庭首页」viewMode，聚合显示：
- 最近 5 条工作区说说（时间线摘要）
- 截止日期 < 3 天的共享任务
- 最近编辑的共享笔记
- 家庭成员状态（在线/离线）

#### 🔴 问题 2：工作区创建与邀请对家庭用户太重

当前工作区需要创建 → 生成邀请码 → 被邀请方在设置中输入 6 位码加入。对家庭用户：

**建议：**
- **一键创建家庭空间**：新增"创建家庭空间"快捷入口，自动设置默认功能开关（开启动态、任务、笔记）
- **扫码/链接加入**：支持分享链接（含一次性 token）加入，无需手工输入邀请码
- **自动发现**：同一内网通过 mDNS 自动发现其他家庭成员
- **预置角色模板**：家庭模板（所有人 editor）、亲子模板（家长 admin，孩子 editor/commenter）

#### 🟡 问题 3：家庭共享缺乏「通知中心」

当前有 @提及通知，但没有共享笔记的变更通知、说说评论通知、任务完成通知。

**建议：** 扩展通知类型：
- 共享笔记被修改的通知
- 说说被评论/点赞的通知（实现评论区 + 点赞功能）
- 待办被标记完成的通知
- 聚合通知中心（不限于 @提及）

### 2.2 开箱体验优化

#### 🟡 问题 4：首次启动引导缺失

当前空数据库启动直接进入笔记列表，没有引导。

**建议：** 首次运行向导：
- 步骤 1：创建家庭空间 / 加入现有家庭
- 步骤 2：邀请成员（展示邀请码或链接）
- 步骤 3：创建第一个笔记本（"家庭备忘录"）
- 步骤 4：选择 AI 配置（可选）或跳过

#### 🟡 问题 5：数据备份对家人不够透明

当前备份在设置深处配置。

**建议：** 在家庭空间首页显示备份状态（上次备份时间、自动备份是否开启），家庭成员一目了然。

### 2.3 功能减法建议

以下功能对 3-4 人家庭场景属于过度设计，可考虑隐藏或默认关闭：

| 功能 | 原因 |
|------|------|
| AI 自定义提示词模板管理 | 普通用户不会用到 |
| API 令牌管理 | 开发者功能 |
| Webhook 配置 | 极客功能 |
| 云端/本地模式切换 | Electron 桌面端专属，家庭以 Web 为主 |
| 2FA | 过度防护，家庭场景密码足矣 |
| 管理员用户管理/工作区管理 | 家庭 3-4 人不需要复杂管理 |

## 三、技术架构优化建议

### 3.1 架构层面的核心风险

#### 🔴 问题 1：单体状态管理将成为瓶颈

`AppContext` 使用单个 `useReducer` 管理所有全局状态（22 个字段），包括笔记列表、活动笔记、搜索、侧栏配置等。随着功能增加：

**风险：**
- 任何 dispatch 都会导致所有 `useApp()` consumer 重渲染（即使只关心 tags 变化，也会收到 notebooks 更新）
- 当前靠 `useMemo` + `React.memo` 缓解，但不是根本解决
- 22 个 action types 在一个大型 switch 中维护

**建议：**
- 将状态拆分为领域子 context：
  ```
  AppContext        → UI 状态（sidebar, viewMode, mobile）
  NotesContext      → 笔记列表、活动笔记、笔记刷新 token
  CollaborationContext → 工作区、成员、实时协作状态
  NotificationContext  → 通知、未读数
  ```
- 或渐进式引入 Zustand（~2KB），原生支持 selector 避免不必要渲染
- **推荐 Zustand**：与当前 useReducer 模式可以共存，迁移成本低，selector 精确控制渲染

#### 🔴 问题 2：api.ts 类型安全存在漏洞

`api` 对象在 3106 行的大对象中定义，但 `mentions` 等新添加的 API 方法是**运行时追加**（通过 head/tail 拼接），TypeScript 类型系统无法感知：

```typescript
// 当前 workaround（见 MentionList.tsx）
const api = _api as any;
```

**风险：** 调用链完全无类型检查，param 变更不报错。

**建议：**
- 将 `api` 对象定义为完整的类型接口，确保增量添加的方法是类型安全的
- 或者拆分为独立模块（`api/notes.ts`、`api/diary.ts`、`api/mentions.ts`），每个导出类型安全的函数簇
- 推荐渐进式：先在 `types/index.ts` 中定义 `ApiClient` 接口，再逐一实现

#### 🟡 问题 3：SQLite 单点写入 + Docker 部署的并发隐患

当前使用 `better-sqlite3`（同步、单连接）。单容器部署没有问题，但注意：

- WAL 模式允许多读一写，但所有写操作串行
- 前端离线队列（offlineQueue）在并发场景下可能出现写入顺序错误
- sqlite-vec 是自定义扩展，升级兼容性需要关注

**建议：**
- 对家庭 3-4 人，SQLite 完全够用（每日写入量 < 1000 条），**暂时不需要迁移到 PostgreSQL**
- 添加 WAL 自动检查点监控：当日志过大时触发 `wal_checkpoint(TRUNCATE)`
- 离线队列添加**冲突检测**：写入前对比 `updatedAt`，服务端版本新时丢弃本地

### 3.2 构建与性能

#### 🔴 问题 4：前端包体积过大

当前依赖了 Tiptap + CodeMirror + Mermaid + KaTeX + Framer Motion + Lucide Icons，粗略估计：

| 依赖 | 压缩后估算 |
|------|-----------|
| @tiptap/* + prosemirror | ~300KB |
| @codemirror/* | ~200KB |
| framer-motion | ~30KB |
| mermaid | ~200KB |
| katex | ~60KB |
| lucide-react（全量） | ~100KB |
| marked + dompurify | ~30KB |
| **合计（gzip）** | **~700KB+** |

**风险：** 初始加载慢、移动端尤其明显。

**建议：**
- **懒加载编辑器**：TiptapEditor 和 MarkdownEditor 使用 `React.lazy()` + `Suspense`，非活跃编辑器不加载
- **动态导入 Mermaid、KaTeX、PDF.js**：只有有关联内容时才加载
- **Lucide Icons 按需引入**：当前 `import { ... } from "lucide-react"` 多数打包工具仍会全量引入，配置 Tree-shaking
- 使用 `vite-plugin-imagemin` 压缩图片

#### 🟡 问题 5：CSS 变量 + Tailwind 的换肤系统性能

当前 CSS 变量定义在两个完整皮肤（default + macOS）的 light/dark 各 100+ 变量上，全部在 `:root` 中声明。存在冗余定义。

**建议：**
- 减少 CSS 变量数量（部分变量值相同，可合并）
- 使用 `@apply` 提取常用组合，减少模板中的长类名字符串
- 考虑 `unocss` 替代 Tailwind，按需生成更小 CSS

### 3.3 可维护性

#### 🟡 问题 6：组件粒度与复杂度

几个巨型组件：

| 组件 | 行数 | 责任 |
|------|------|------|
| TiptapEditor | 4449 | 编辑器 + 工具栏 + 扩展 + 斜杠菜单 |
| NoteList | 3233 | 列表 + 排序 + 筛选 + 拖拽 + 右键菜单 |
| EditorPane | 2906 | 标题栏 + 编辑器切换 + 属性面板 + 附件 + 版本历史 |
| Sidebar | 2238 | 工作区 + 搜索 + 笔记本树 + 标签 |
| SettingsModal | 1552 | 8 个设置选项卡合并在一个模态框 |

**建议：**
- 按责任拆分子组件（如 Toolbar、ExtensionManager、EditorShell）
- 将巨型文件的纯逻辑部分提取到 hooks 中
- 为每个设置选项卡创建独立组件，SettingsModal 只做选项卡容器

#### 🟡 问题 7：测试覆盖

**现状：** 只有少量测试文件。

**建议：**
- API 层：对 `backend/src/lib/` 下的工具函数增加单元测试（mentions.ts、auth-security.ts）
- 路由层：对 diary、notes、tasks 的关键流程进行 e2e 测试（使用 `supertest` 测试 Hono 路由）
- 前端：对 API 调用层（api.ts）和状态管理层（AppContext）增加简单测试

### 3.4 Docker 部署优化

#### 🟡 问题 8：多容器管理与资源占用

当前 3 个服务（nowen-note + ollama + sensevoice）对家庭 NAS/小主机压力：

| 服务 | 内存估算 | 存储估算 |
|------|---------|---------|
| nowen-note | ~80MB | ~100MB |
| ollama (Qwen2.5:1.5b) | ~1.5GB (CPU) | ~1GB 模型 |
| sensevoice | ~800MB (Python) | ~1GB 模型 |
| **合计** | **~2.4GB** | **~2.1GB** |

**建议：**
- **Ollama/SenseVoice 按需启动**：在 compose 中不设置 `restart: unless-stopped`，改为手动按需启动
- 或者提供轻量模式：仅 nowen-note + 选择性启动 AI 服务
- **模型缓存共享**：sensevoice 模型缓存挂载为命名卷（已做），但 ollama 的模型也在命名卷中（已做）

## 四、UI/UX 优化建议

### 4.1 信息架构（IA）优化

#### 🔴 问题 1：12 种 viewMode 让导航过载

当前导航通过 NavRail 呈现 8 个基本项目（见 NavRail.tsx 第 180-230 行），但实际有 12 种 viewMode。用户需要理解：

- 哪些是笔记列表的变体（all / notebook / favorites / trash / tag / search）
- 哪些是独立模块（tasks / diary / mindmaps / ai-chat / files / mentions）

**建议：**
- **分组导航**（NavRail 已部分做到，但视觉区分不够）：
  ```
  ┌─────────────┐
  │ 📝 所有笔记  │  ← 浅色底
  │ ⭐ 收藏     │
  │ 🗑️ 回收站   │
  │─────────────│
  │ 📖 说说     │  ← 彩色图标
  │ ✅ 待办     │
  │ 🧠 思维导图  │
  │ 🤖 AI 聊天  │
  │ 📎 文件     │
  │─────────────│
  │ 🔔 消息     │
  │ ⚙️ 设置     │
  └─────────────┘
  ```
- **搜索替代导航**：Cmd-K 全局搜索已经实现了，但需要更好的搜索结果分类

#### 🔴 问题 2：笔记本树与 viewMode 的关系不直观

当前选择笔记本（sidebar 中点击）→ viewMode 变为 "notebook"。但切换到"说说"再切回来，不会自动回到上次的笔记本。

**建议：** 恢复最近选中的笔记本状态（已在 AppContext 中存在 `selectedNotebookId`，但跨模块切换时没有记忆）

### 4.2 移动端体验

#### 🟡 问题 3：移动端编辑器体验

Tiptap 编辑器在移动端的工具栏是水平滚动的，操作不够便捷。

**建议：**
- 固定常用操作（B/I/U/标题）在屏幕底部（类似 Notion 移动版）
- 插入操作（图片/表格）改为长按或 + 菜单
- 利用 Capacitor 的 `useKeyboardLayout`（已有）进一步精确适配键盘弹出

#### 🟡 问题 4：说说发布流程可简化

当前发布说说需要填写文本 → 选心情 → 加图片 → 选标签 → 设置可见性 → 点击发布。对于家庭用户记录生活瞬间，流程偏长。

**建议：**
- **快捷说说**：在 NavRail 或首页常驻"随手拍"入口，一键拍照→发布（无需文本）
- **免标签**：家庭可见性默认公开（工作区内），移除可选标签步骤
- **语音优先**：语音消息自动转文字（当前已有转写能力，但作为隐藏的长按手势）

### 4.3 视觉与交互细节

#### 🟡 问题 5：编辑器双模式的认知负担

当前用户可以从 MD 编辑器和 Tiptap 富文本编辑器之间切换（通过 EditorPane 顶部的切换按钮）。对于家庭用户：

- 切换行为不可逆（可能导致格式丢失）
- 两个模式的工具栏不同，学习成本翻倍
- Yjs 协作在 MD 模式下不可用

**建议：**
- **默认隐藏 MD 切换**：对普通用户只展示 Tiptap 富文本编辑器
- 在设置中提供「启用 Markdown 模式」开关
- 切换时展示格式兼容性警告

#### 🟢 问题 6：说说内容渲染优化

当前说说（DiaryCenter）使用 `marked` + `DOMPurify` 渲染 Markdown。但说说的定位是「朋友圈风格」短内容（诗、心情、段子），Markdown 渲染过于"笔记感"。

**建议：**
- 说说编辑器可以改为纯文本 + Emoji 选择器（已有心情 emoji）
- 或提供"纯文本模式"切换，关闭 Markdown 渲染
- 支持更丰富的排版：对齐方式、字体大小、背景色块（类似微博/Instagram）

#### 🟢 问题 7：待办日历可交互性

当前 TaskCalendar 和 DiaryCalendar 点击日期后只是切换回列表视图，没有高亮选中日期或跳转到当天的内容。

**建议：**
- 点击日期后保持日历可见，在下方展开当天的条目列表（类似 Google Calendar）
- 或者点击日期跳转到列表模式时，自动按该日期过滤（目前 diary 已做，task 需要补）

### 4.4 无障碍与包容性

#### 🟡 问题 8：主题一致性

当前有两个皮肤（default + macOS）各有一套 light/dark 变量，但：
- 部分组件硬编码颜色（如 `text-accent-primary` 可能在某些自定义颜色下不可读）
- macOS 皮肤与默认皮肤的差异主要是标题栏拖拽区域

**建议：** 统一使用 CSS 变量，消除硬编码颜色值。可供选择的主题增加"高对比度"主题（方便年长家庭成员）。

## 五、安全与合规建议

### 5.1 家庭场景的隐私保护

#### 🟡 建议 1：本地优先架构

当前所有数据存储在服务器端（SQLite），家庭成员的数据在同一数据库中。

**建议：**
- 每个家庭成员的个人笔记默认仅自己可见（当前已通过 `workspaceId IS NULL` 实现）
- 共享内容在家庭工作区中存储
- 管理员（家长）不应默认拥有查看其他成员私人笔记的权限

#### 🟡 建议 2：附件 URL 的安全性

当前附件通过 `GET /api/attachments/:id` 无认证访问（依赖 UUID 不可猜测）。在家庭内网场景中风险较低，但如暴露到公网：

**建议：** 增加可选的 referer 校验或短期签名 URL（对公网部署场景）

## 六、项目治理建议

### 6.1 上游同步策略

**问题：** 当前基于 NowenNote 开源项目二次开发，上游持续更新。

**建议：**
- 保持 git 历史可追踪，标记分叉点
- 对核心库（如 Tiptap、Hono、better-sqlite3）的升级跟上上游
- 对业务逻辑的修改（如家庭功能增强）独立为模块，减少与上游的冲突
- 定期（每月）合并上游更新

### 6.2 测试策略

**当前测试覆盖：极少**
- 后端：无正式的 jest/vitest 测试文件（发现一个 importMemos 测试文件）
- 前端：无正式的组件测试
