# 统计模块仪表盘重设计

## 目标

让统计页在不新增接口和第三方图表依赖的前提下，优先呈现用户最关心的完成进度、待处理风险和最近动态，解决现有页面卡片同权、留白过多、动态列表难扫读的问题。

## 方案比较

1. 保持现有布局，仅优化颜色和圆角：改动小，但无法改善信息层级。
2. 重构为数据仪表盘（采用）：首屏增加完成率主视觉、风险摘要和活动时间线；保留全部原有标签页与数据来源。
3. 引入完整图表库：视觉表现更强，但增加依赖、数据口径与维护成本，超出本次范围。

## 信息架构与视觉

- 顶部使用低饱和渐变摘要区：标题、说明、刷新操作与圆形任务完成率并列；完成率成为首屏视觉锚点。
- 摘要指标使用有语义的浅色背景和图标容器：待办/逾期使用暖色风险提示，习惯与连续天数使用绿色与橙色，避免所有卡片仅以同一种蓝色区分。
- 最近动态使用带图标、时间和辅助说明的行式时间线；无记录时复用明确的空状态，避免大块空白。
- 项目分布改为进度条列表，展示完成数和比例；任务记录、趋势、习惯周/月/年视图统一卡片边框、标题和内容密度。
- 不改变现有统计计算、路由、请求及标签页交互，保证数据口径不变。

## 组件边界

- `StatsCenter` 继续负责数据聚合和标签页状态。
- 提取展示型 `MetricCard`、`ActivityPanel`、`SectionCard` 等局部组件，只接收显示数据，不发起请求。
- 动态、项目分布与空状态由现有聚合结果驱动，不新增后端 API。

## 验证

- 为首屏摘要区和项目完成进度补充 DOM 回归断言。
- 运行 `StatsCenter` 相关 Vitest 测试。
- 运行前端构建；若出现与本次无关的既有错误，单独记录。

## 自检

- 无待定项、占位符或未定义的设计决策。
- 不改变数据源和行为，只改善界面层级与可读性。
- 范围限于统计模块，可由单个实现计划覆盖。
