# 删除任务后排除统计动态设计

## 背景与目标

任务统计使用 `task_activity_events` 保存创建和完成事件。该表是追加式历史流水，任务被删除后事件仍然存在；当前活动接口直接读取流水，导致“最近任务动态”、任务记录、趋势和热力图继续展示已删除任务。

目标是：任务删除后，该任务及其级联删除子任务的活动不再出现在任何用户可见的任务统计中，同时不物理删除底层历史流水。

## 方案

在后端任务活动查询接口统一过滤无对应任务的流水。查询以 `task_activity_events` 为主表，并增加关联存在性条件：只有 `tasks` 表中仍存在相同 `id` 的任务，活动才可以返回。

选择接口层过滤而不是前端过滤或删除流水，原因如下：

- 所有统计视图共享同一活动接口，可以保持最近动态、记录、趋势和热力图口径一致。
- 单个删除、批量删除以及父任务级联删除都会让任务行消失，因此无需在多个删除入口重复清理逻辑。
- 保留历史流水，不改变已有迁移和审计语义，也不需要数据迁移。

## 数据流

1. 创建或完成任务时，现有数据库触发器继续写入 `task_activity_events`。
2. 删除任务时，现有任务删除逻辑保持不变；任务行及级联子任务行从 `tasks` 消失。
3. 前端加载统计页时请求任务活动接口。
4. 后端只返回仍能在 `tasks` 中找到对应任务的活动。
5. 前端继续用同一活动集合生成最近动态、记录、趋势和热力图，因此已删除任务在全部统计视图中消失。

## 查询与权限

活动查询保留现有用户、工作区、日期范围和数量限制。新增条件只负责验证活动关联的任务仍存在，不放宽现有作用域权限。

SQL 中为活动表使用别名，并限定所有作用域、时间和排序字段，避免与 `tasks` 的同名字段产生歧义。存在性检查只按任务主键匹配；任务 ID 在系统内唯一，原有活动作用域条件继续承担权限隔离。

## 兼容性与非目标

- 不删除或迁移既有 `task_activity_events` 数据。
- 不改变任务删除 API 的响应格式。
- 不修改前端统计组件和统计文案。
- 不恢复已删除任务，也不新增“显示已删除任务历史”的开关。

## 测试与验收

新增后端回归测试覆盖以下行为：

1. 存在任务的创建、完成活动可以由统计接口返回。
2. 删除任务后，对应活动仍保留在底层流水表中。
3. 删除任务后，统计活动接口不再返回对应活动。
4. 其他未删除任务的活动仍正常返回。

验收命令包括新增回归测试、相关任务活动测试、后端 TypeScript 构建和前端生产构建。前端无需代码改动，但生产构建用于确认共享 API 类型和现有统计页面未受影响。
