# 多端同步修复验证清单

本文档用于 `1.1.7` hotfix 发版前验证 PC/Web 与移动端同账号同步行为。

## 验证环境

建议至少准备两个客户端：

- 客户端 A：PC 桌面端或 Web 端。
- 客户端 B：手机端或移动 WebView。
- 两端登录同一个账号，连接同一个服务端。
- 测试前确认两端都能正常打开同一篇笔记。

## 1. 列表实时同步

步骤：

1. 客户端 B 停留在笔记列表，不打开目标笔记。
2. 客户端 A 打开目标笔记，修改标题和正文首段。
3. 等待自动保存完成。

期望：

- 客户端 B 的列表卡片标题、摘要、更新时间自动更新。
- 不需要切换视图、刷新页面或重新进入应用。

## 2. 当前笔记自动刷新（无本地修改）

步骤：

1. 客户端 A 和客户端 B 同时打开同一篇笔记。
2. 客户端 B 不输入、不聚焦编辑器，保持只读浏览状态。
3. 客户端 A 修改正文并保存。

期望：

- 客户端 B 自动拉取远端最新版并刷新当前编辑器内容。
- 不出现“同步失败”或持续旧内容。
- 不需要离开当前笔记再切回来。

## 3. 当前笔记冲突保护（本地正在编辑）

步骤：

1. 客户端 A 和客户端 B 同时打开同一篇笔记。
2. 客户端 B 在编辑器中输入一段未保存内容，保持编辑状态。
3. 客户端 A 修改同一篇笔记并保存。

期望：

- 客户端 B 不会被远端内容静默覆盖。
- 客户端 B 显示远端更新冲突提示。
- 本地未保存内容仍保留在编辑器中。

## 4. 冲突操作：重新加载远端版本

前置：已触发第 3 节冲突提示。

步骤：

1. 在客户端 B 点击“重新加载”。

期望：

- 客户端 B 丢弃本地 pending 修改。
- 客户端 B 加载客户端 A 保存的远端最新版。
- 后续继续编辑能正常保存，不再连续 409。

## 5. 冲突操作：覆盖远端

前置：重新制造一次第 3 节冲突。

步骤：

1. 在客户端 B 点击“覆盖远端”。
2. 等待保存完成。
3. 观察客户端 A。

期望：

- 客户端 B 当前编辑器内容成为服务端最新版。
- 客户端 A 收到远端更新并刷新或提示。
- 双端最终版本一致。

## 6. 移动端后台恢复补偿

步骤：

1. 客户端 B 打开目标笔记。
2. 将客户端 B 切到后台。
3. 客户端 A 修改目标笔记并保存。
4. 将客户端 B 切回前台。

期望：

- 客户端 B 回前台后主动检查当前笔记版本。
- 无本地修改时自动刷新。
- 有本地修改时进入冲突提示。

## 7. 断网恢复补偿

步骤：

1. 客户端 B 打开目标笔记。
2. 断开客户端 B 网络。
3. 客户端 A 修改目标笔记并保存。
4. 客户端 B 恢复网络。

期望：

- 客户端 B 恢复网络后主动检查当前笔记版本。
- 不静默覆盖本地内容。
- 若存在本地修改，应提示冲突并保留草稿。

## 8. 自回声检查

步骤：

1. 单独使用客户端 A 打开并编辑笔记。
2. 连续输入多段内容。

期望：

- 客户端 A 不应因为自己的保存广播弹出远端更新提示。
- 光标和输入不应出现明显回退。
- 保存状态应正常从“保存中”变为“已保存”。

## 9. 发版门禁

发版前必须通过：

```bash
npm run build:backend
npm run build:frontend
```

同时检查：

```bash
git --no-pager diff --check
```

验收标准：

- 构建无错误。
- `diff --check` 无输出。
- 上述 8 个场景通过。
